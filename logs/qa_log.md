### Q1: 作業ブランチの大きさに基準はあるか

`#ブランチ` `#PR` `#レビュー`

**質問**: 作業ブランチの役割の大きさは基準がありますか？

**回答**:

明確な数値基準は組織によるが、広く共有されている目安がある。

| 指標 | 推奨 | 避けるべき |
|------|------|-----------|
| 変更行数 | ~300行以下 | 1000行超 |
| 作業期間 | 数時間〜2日 | 1週間以上 |
| 目的の数 | 1つ | 複数を混ぜる |

**最も重要なルール**: **1ブランチ = 1つの目的**

```
✅ 良い例: feature/add-subtract-function（目的が明確）
❌ 悪い例: feature/various-improvements（何が入ってるか不明）
```

**AI駆動開発では**: AIへの指示も小さく分けることで、レビューしやすいPRを作れる。

### Q2: mainが先に進むとコンフリクトリスクが増大するとは

`#コンフリクト` `#ブランチ寿命`

**質問**: 「ブランチの作業中にmainがどんどん先に進んでいる＝コンフリクトのリスク増大」の意味を教えて。

**回答**:

自分が作業ブランチで作業している**間に**、他メンバーのPRが `main` にマージされて main が先に進む状況。

```
ブランチを切った時点:
main:     ●───●───●
                   \
feature:            ●

1週間後（まだ作業中）:
main:     ●───●───●───●───●───●───●   ← 他の人のPRが5つマージされた
                   \
feature:            ●───●───●         ← 自分の作業

→ main の変更と自分の変更が同じ箇所を触っていたら 💥 コンフリクト！
```

**対策**: 作業を小さく区切ってこまめにマージする。ブランチの寿命が短いほどコンフリクトしにくい。

### Q3: ブランチ名の `feature/` とは

`#命名規則` `#ブランチ`

**質問**: ここでいう feature とはどんな意味がある？

**回答**:

ブランチの**目的を示す接頭辞（プレフィックス）**。Git の機能ではなく、チームの命名規則（慣習）。

| プレフィックス | 意味 | 例 |
|--------------|------|-----|
| `feature/` | 新機能の追加 | `feature/add-subtract` |
| `fix/` | バグ修正 | `fix/login-crash` |
| `docs/` | ドキュメント修正 | `docs/update-readme` |
| `refactor/` | コード整理（動作は変えない） | `refactor/cleanup-utils` |
| `chore/` | 雑務（設定変更など） | `chore/update-dependencies` |

> Step 6 のコミットメッセージ規約（`feat:`, `fix:`）と対応させると一貫性が出る。

### Q4: AIに小さく指示すると開発速度が落ちないか？大きな機能はどうする？

`#AI駆動開発` `#ブランチ戦略` `#大規模変更`

**質問**: AIコード生成を小さく区切ると、一気に開発を回せなくならない？ 大きな機能の追加や大幅な修正があるときはどうすべき？

**回答**:

**結論: AIにはガンガン書かせてOK。「コミットとPRの分け方」で整理する。**

| 戦略 | やり方 | 向いている場面 |
|------|--------|--------------|
| **戦略1** | 1つのPR内でコミットを目的別に小分け | 中規模の変更（~500行） |
| **戦略2** | PR自体を複数に分ける（Stacked PRs） | 大規模（1000行超） |
| **戦略3** | フィーチャーフラグで新旧コードを共存 | 大幅リファクタリング |

**大事な考え方**: ボトルネックはコード生成ではなく**レビューと品質保証**。小分けの目的は「AIを遅くする」ためではなく「**人間のレビューを効率化する**」ため。

### Q5: 戦略1（コミット小分け）と戦略2（PR分割）の違い

`#戦略比較` `#PR` `#コミット`

**質問**: 戦略1と2の違いを分かりやすく説明して。

**回答**:

一言でいうと:
- **戦略1** = 1つのPRの中でコミットを小分け（段ボール1箱、中身を仕切りで整理）
- **戦略2** = PR自体を複数に分ける（段ボールを複数箱に分ける）

```
戦略1:  PR #1 の中に → コミット1, コミット2, コミット3, コミット4

戦略2:  PR #1（モデル）→ マージ → PR #2（ログイン）→ マージ → PR #3（テスト）→ マージ
```

| | 戦略1 | 戦略2 |
|--|------|------|
| 向いている場面 | 中規模（~500行） | 大規模（1000行超） |
| マージ | 全部揃ってから1回 | 段階的に1つずつ |
| 途中で方向転換 | PR全体やり直し | 途中のPRまでは活かせる |
| 使用頻度 | ★★★ よく使う | ★★ 大きな機能で使う |

**ほとんどの場合は戦略1で十分。**

### Q6: 戦略1でコミット1を修正したら他のコミットに影響は出るか

`#コミット修正` `#rebase` `#squash`

**質問**: 戦略1でコミット1の修正が必要になったら、ほかのコミットに影響は出ませんか？

**回答**:

**影響が出る場合と出ない場合がある。**

- **影響なし**: 各コミットが別ファイル・別箇所を触っている場合 → 新しいコミットとして修正を積めばOK
- **影響あり**: コミット1で作った関数をコミット2以降で使っている場合 → 関数名を変えたら後が壊れる

**実務での対処法**:

| 方法 | いつ使う |
|------|---------|
| ① 新しいコミットとして修正を積む（推奨） | ほとんどの場合これでOK |
| ② `git rebase -i` で過去コミットを書き換え | PR前に履歴をきれいにしたい時 |
| ③ マージ時に squash で全部まとめる | GitHub上でマージする時に選べる |

> 💡 実務では **③ squash マージ** が非常によく使われる。作業中のコミットが多少汚くてもOK。（Step 5 で体験予定）

```
作業中（ブランチ内）:              マージ後（mainに入る）:
  コミット1: モデル追加
  コミット2: ログイン追加    →    1つのコミット:「ユーザー認証機能を追加」
  コミット3: やっぱモデル修正
  コミット4: テスト追加
```

### Q7: mainが先に進んでしまった時はどうすればいいか

`#コンフリクト` `#merge` `#rebase` `#mainとの同期`

**質問**: mainがどんどん先に進んでいる時は、どうすればいいのでしょうか？

**回答**:

**やるべきこと: 「mainの最新を自分のブランチに取り込む」。** 2つの方法がある。

**方法1: `git merge main`（安全・初心者向け）**

```
main:     ●───●───●───●───●
                   \         \
feature:            ●───●───M  ← mainの変更を取り込む「マージコミット(M)」ができる
```
- ✅ 安全。履歴がそのまま残る
- ❌ マージコミットが増えて履歴がやや複雑になる

**方法2: `git rebase main`（履歴がきれい・中級者向け）**

```
main:     ●───●───●───●───●
                             \
feature:                      ●'──●'  ← mainの最新の先にコミットが「付け替え」られる
```
- ✅ 履歴が一直線できれい
- ❌ コミットが「書き換わる」ので、既にpush済みの場合は注意が必要

**どちらを使うべきか:**

| 状況 | 推奨 |
|------|------|
| まだ push していない | どちらでもOK（rebase がきれい） |
| 既に push 済み・他の人も見ている | `merge` が安全 |
| よく分からない | **`merge` を使えば間違いない** |

**そもそも取り込む必要があるかの判断:**

| 状況 | どうする |
|------|------|
| main の変更が自分の作業に**関係ない** | 何もしなくてOK。PR作成時にGitHubが自動判断 |
| main の変更が**同じファイルを触っている** | 早めに取り込む（コンフリクトが小さいうちに対処） |
| PRに「コンフリクトあり」と表示された | その時点で取り込んで解決する |

> 💡 **実務のコツ**: 毎朝 `git switch feature/xxx && git merge main` を習慣にすると、差が広がりすぎる前に取り込める。Step 5 で実際に体験予定。

### Q8: `git rebase main` がよくわからない

`#rebase` `#履歴整理` `#merge比較`

**質問**: 方法2: `git rebase main`（履歴がきれい・中級者向け）がよくわからないです。

**回答**:

rebase は「re（再び）+ base（土台）」= **土台を置き直す**という意味。自分のコミットの分岐点を、mainの最新地点に「引っ越し」させる操作。

```
rebase 前:
main:     旧駅───A───B───新駅
                  \
feature:           ①───②         ← 「旧駅」から分岐

rebase 後:
main:     旧駅───A───B───新駅
                              \
feature:                       ①'──②'   ← 「新駅」から分岐し直した
```

**なぜ ①' ②' になるか**: 元の①②は「旧駅の時点のコード」を前提に書かれていたが、rebase後は「新駅の時点のコード」が前提になる。Gitが内部的にコミットをコピーして作り直すため、コミットIDが変わる。

**merge との結果比較**:

```
【merge】分岐と合流の記録が残る    【rebase】一直線になる
 旧駅                               旧駅
  │                                  │
  A                                  A
  │                                  │
  B                                  B
  │                                  │
 新駅                               新駅
  │ \                                │
  │ 合流(M)                         ①'
                                     │
                                    ②'
```

**鉄則**: rebase は「まだ自分しか知らないコミット」に対して使う。push済みのコミットを rebase すると他の人の手元と食い違いが起きる。Step 5 で詳しく扱う。

### Q9: mergeとpullは何が違うのか

`#merge` `#pull` `#fetch` `#基礎概念`

**質問**: mergeをちゃんと理解できていないかもしれません。pullと何が違うの？

**回答**:

**`git pull` = `git fetch` + `git merge`。** pull は merge を**含んでいる**操作。

| 操作 | やること | 例えると |
|------|---------|--------|
| `fetch` | リモート（GitHub）の最新情報をダウンロード。作業ファイルには影響なし | 郵便受けを確認 |
| `merge` | 別ブランチの変更を今のブランチに合流させる | ノートに書き写す |
| `pull` | fetch + merge を一気にやるショートカット | 郵便受けから取って即反映 |

**merge の対象による違い:**

| コマンド | 何を | どこに | 場面 |
|---------|------|--------|------|
| `git pull origin main` | **GitHub上の** main を | 今のブランチに | リモートの変更を取得 |
| `git merge main` | **ローカルの** main を | 今のブランチに | ブランチ間の合流 |

**よくある実務のパターン:**
```
git switch main
git pull origin main        ← まず main を最新にする（GitHub → ローカル main）
git switch feature/xxx
git merge main              ← 最新の main を feature に取り込む（ローカル main → ローカル feature）
```

### Q10: stashとは何か

`#stash` `#一時退避` `#基礎概念`

**質問**: stashとは何？

**回答**:

`git stash` は**作業中の変更を一時的に退避させる**機能。「机の上の書類を引き出しにしまう」イメージ。

```bash
git stash           # 編集内容を「引き出し」にしまう → ファイルが編集前に戻る
git switch main     # 安心してブランチを切り替えられる
# ... main で作業 ...
git switch feature/xxx
git stash pop       # 「引き出し」から取り出す → 編集途中の状態に戻る
```

| 操作 | 例えると |
|------|--------|
| `git stash` | 机の上の書類を引き出しにしまう |
| `git stash pop` | 引き出しから書類を机に戻す |

> Worktree を使えば stash が不要になるケースが多い。Step 3 で詳しく扱う。

### Q11: 同時編集とは一人が複数ブランチを編集できるという意味か

`#worktree` `#同時編集`

**質問**: ここでいう同時編集とは、複数人が同じフォルダを操作できるでなく、一人が複数ブランチを編集できるで認識があっている？

**回答**:

**その認識で完全に合っている。**

```
❌ 複数人が同じフォルダを操作できる
✅ 一人が複数ブランチを同時に開いて編集できる
```

具体的には VSCode ウィンドウを2つ開いて、それぞれ別フォルダ（別ブランチ）を同時に編集できる。

複数人の共同編集は Git の機能ではなく、各自が自分のPCにリポジトリをクローンして、各自のブランチで作業 → PR でマージ（GitHub Flow）。

### Q12: pushでローカルのブランチがGitHubに登録された認識であっているか

`#push` `#リモート` `#ブランチ`

**質問**: ╄ Worktree側でコミット → push、ここではローカルにしかなかったブランチが、github側に登録されたで認識はあってる？

**回答**:

**完全に合っている。**

```
push 前: ローカルにだけ feature/add-subtract がある
push 後: GitHubにも feature/add-subtract が作られる
```

`git push -u origin feature/add-subtract` の各パーツ:
- `push`: ローカルの内容をGitHubに送る
- `-u`: ローカルブランチとリモートブランチを紐付ける（次回から `git push` だけで済む）
- `origin`: GitHubのリポジトリの名前
- `feature/add-subtract`: 送るブランチ名

### Q13: このPRはマージしておいたほうがいいか

`#PR` `#merge` `#クリーンアップ`

**質問**: ここでPRは、mergしといたほうがいい？

**回答**:

**はい、マージすべき。** 理由:
- 残しておくと以降の Step で main と feature が違う状態のまま進み混乱の元
- PRを通じたマージ自体が GitHub Flow の実践
- マージ後の Worktree クリーンアップ手順も体験できる

実際に squash merge + ブランチ削除 + `git pull` + `git worktree remove` まで実行した。

### Q14: stash向きの具体的な作業とは

`#stash` `#worktree` `#使い分け`

**質問**: 実務の目安: 5分以内で戻れるなら stash、とあるが、具体的にはどのような作業が対象？

**回答**:

**stash 向き（ちょっと見て戻ってくる）:**
- コードの確認（「mainのあの関数どう書いてあったっけ？」）
- 動作確認（「本番の状態でバグ再現するか試したい」）
- 設定のコピー（「別ブランチの設定ファイルを参考にしたい」）
- 急な1行修正（「READMEのタイポ直して」）

→ 共通点: **切り替え先で「見る」か「ほんの少し直す」だけ**

**Worktree 向き（しばらく別作業）:**
- 緊急バグ修正（調査・修正・テストで30分～）
- コードレビュー（他の人のPRをビルドして動作確認）
- 並行開発（機能Aのビルド待ちの間に機能Bを進める）
- 比較しながらの作業（旧新バージョンを並べて見比べる）

> 💡 Worktreeを覚えると stash をあまり使わなくなる人も多い。stash は「知っているけど場面を選ぶ」くらいの位置づけでOK。

### Q15: mainで修正した内容は元のブランチに反映されるか

`#merge` `#自動マージ` `#ブランチ間の同期`

**質問**: 急な1行修正など main を修正した場合、もとに戻ったブランチでは修正内容を反映させますか？

**回答**:

**自動では反映されない。だがほとんどの場合、何もしなくてOK。**

| 状況 | どうする |
|------|------|
| main の修正が自分の作業に**関係ない** | 何もしない。PRマージ時にGitが自動で両方を合流してくれる |
| main の修正が自分の作業に**影響する** | `git merge main` で取り込む |

> 「わざわざ取り込まなくても、最終的にマージする時に自動で合流する」のが Git の賢いところ。同じ箇所を触っている場合だけコンフリクトになる。

### Q16: コンフリクトはどういう条件で発生するか

`#コンフリクト` `#発生条件`

**質問**: コンフリクトの条件を教えて。同じファイルの同じ場所をいじったとき？同じ関数をいじったとき？

**回答**:

**「同じファイルの同じ行」を2つのブランチが違う内容に変えた時。** Gitは「関数」や「意味」を理解しない。純粋に「行」の位置で判断。

| mainの変更 | featureの変更 | コンフリクト？ |
|------------|---------------|-----------|
| 2行目を変更 | 2行目を変更 | 💥 する（同じ行） |
| 2行目を変更 | 5行目を変更 | ✅ しない（違う行） |
| 同じ関数の3行目 | 同じ関数の5行目 | ✅ しない（同じ関数でも違う行ならOK） |

### Q17: commitの意味とpullとの違い

`#commit` `#pull` `#push` `#基礎概念`

**質問**: コミットの意味をあらためて教えて。pullとの違いは？

**回答**:

commitとpullは**方向が真逆**。

| 操作 | 方向 | 例えると |
|------|------|--------|
| `commit` | 自分のPC内でセーブ | ゲームのセーブ |
| `push` | 自分のPC → GitHub | セーブデータをクラウドにアップロード |
| `pull` | GitHub → 自分のPC | クラウドからダウンロード |

全体の流れ: `ファイル編集 → git add（準備） → git commit（セーブ） → git push（アップロード）`

### Q18: git addの意味

`#add` `#staging` `#基礎概念`

**質問**: git addの意味は？

**回答**:

**「次のコミットに含めるファイルを選ぶ」操作。** 変更したファイルのうち「これだけコミットしたい」と選ぶために存在する。

```
作業ディレクトリ        ステージ            リポジトリ（履歴）
 (ファイル編集)       (コミット準備中)    (セーブ済み)

  app.py     ──add──→  app.py     ──commit──→  記録！
  memo.txt              (選ばなかった)        (入らない)
```

| 操作 | 例えると |
|------|--------|
| ファイル編集 | 机の上で書類を書く |
| `git add` | 「この書類を提出する」と封筒に入れる |
| `git commit` | 封筒を閉じて提出（確定） |
| `git push` | 提出した封筒を郵便で送る（GitHubへ） |

### Q19: ブランチを作成してから実装を始めるのか

`#ブランチ` `#ワークフロー`

**質問**: そもそもブランチを作成してから実装作業を開始する？それともコミットするときにブランチを作成するの？

**回答**:

**先にブランチを作ってから作業を始めるのが基本。**

```
✅ 正しい順番: git switch -c feature/xxx → 編集 → commit
❌ やりがち: mainのまま編集を始めてしまう
```

もし main で編集を始めてしまった場合、まだコミットしていなければ `git switch -c feature/xxx` で編集中の変更を持ったまま新ブランチに移れる。

> 「何か作業を始めよう」と思ったら `git switch -c` から始める。条件反射にする。

### Q20: 分岐がなくなることのメリットは

`#rebase` `#履歴` `#squash`

**質問**: 分岐がなくなることのメリットは？

**回答**:

主にチーム開発で効く。

| メリット | 説明 |
|---------|------|
| 履歴が読みやすい | `git log` で上から順に追える |
| バグ探しが楽 | `git bisect` で原因コミットを素早く特定できる |
| 差分が見やすい | レビュー時に変更の順番を追いやすい |

**実務の現実**: rebase を自分で使いこなす必要は薄い。GitHub の **squash merge** が同じ効果を自動で得られるので、日常的には squash merge に任せるのがスタンダード。

---

### Q21: squash mergeは合流ではなくまとめて新コミットを作る

`#squash` `#merge` `#履歴`

**質問**: squash merge は合流しているのは変わらないけど、1つのみの中継点になるイメージ？

**回答**:

**squash merge は「合流」ではない。** ブランチの全変更を**コピーして1つにまとめた新しいコミットを main に直接作る**操作。

```
通常 merge:  main: ●─●─M       ← M(マージコミット)で合流。分岐の記録が残る
                      / 
feature:       ①②③

squash merge: main: ●─●─S       ← S = ①②③を全部まとめた新コミット。分岐の痕跡なし
feature:       ①②③  ← mainの履歴には現れない
```

**なぜ人気か**: main の履歴が「PR 1つ = コミット 1つ」でスッキリする。「何番のPRで何が入ったか」が一目瞭然。
