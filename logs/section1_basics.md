## Step 0: 演習用リポジトリの準備

### 概要

Git/GitHub 学習のための演習用リポジトリを作成し、GitHub に連携した。

- `logs/` — 学習ログ（Q&A、コマンド履歴、進捗）

## Step 1: ブランチ戦略（GitHub Flow）

### 概要

GitHub Flow は「main ブランチを常に安定した状態に保ち、全ての変更を作業ブランチ経由で PR を通してマージする」ワークフロー。

### GitHub Flow の流れ

```
1. main から作業ブランチを作る    git switch -c feature/xxx
2. コードを書いてコミットする       git add → git commit
3. GitHub に push する              git push
4. Pull Request（PR）を作る        gh pr create
5. レビュー → CI通過 → マージ     gh pr merge --squash
6. 作業ブランチを削除              --delete-branch で自動削除
```

### ブランチの命名規則

| プレフィックス | 用途 | 例 |
|--------------|------|-----|
| `feature/` | 新機能 | `feature/add-login` |
| `fix/` | バグ修正 | `fix/login-error` |
| `docs/` | ドキュメント | `docs/update-readme` |
| `refactor/` | リファクタリング | `refactor/split-utils` |
| `test/` | テスト追加 | `test/add-unit-tests` |

### Q1: 作業ブランチの役割の大きさに基準はあるか

`#ブランチ` `#PR` `#作業単位`

**質問**: 作業ブランチの役割の大きさは基準がありますか？

**回答**:

明確な数値基準はないが、実務の目安がある。

| 指標 | 目安 |
|------|------|
| 変更行数 | 200〜400行以内が理想（レビューしやすい） |
| 作業期間 | 1日〜数日以内（長引くとコンフリクトリスク増大） |
| 内容の単位 | 「1つの機能」「1つのバグ修正」が1ブランチ |

大きすぎるとレビューが大変で、小さすぎるとPRの数が増えすぎる。「レビュアーが30分以内にレビューできるか？」が目安。

### Q2: mainがどんどん先に進んでいるの意味

`#コンフリクト` `#ブランチ` `#分岐`

**質問**: ブランチの作業中にmainがどんどん先に進んでいる（＝コンフリクトのリスク増大）の意味を教えて。

**回答**:

```
分岐した時点:
main:     ●───●（ここから分岐）
feature:       ├─①

1週間後（他のメンバーのPRがマージされてmainが進んだ）:
main:     ●───●───A───B───C───D（どんどん先へ）
feature:       ├─①──②──③

→ 分岐からの「距離」が遠いほど、同じファイルの同じ箇所を触っている確率が上がる
→ コンフリクトの確率が増える
```

対策: ブランチを短期間で終わらせる。長引く場合は `git merge main` でこまめにmainの変更を取り込む。

### Q3: AIにコード生成を頼むと大量の変更を作りがち

`#AI駆動開発` `#コミット戦略` `#PR`

**質問**: AIにコード生成を頼むと一度に大量の変更を作りがち。「まずこの関数だけ」と指示を小分けにするとAIの効率が落ちませんか？大きな機能の追加や大幅な修正がある時はどうすべき？

**回答**:

AIへの指示の粒度と、Gitのコミット/PRの粒度は**別々に管理できる**。

**戦略1: 1つのPR内でコミットを小分けにする（基本）**
```
AIに全部作らせる → 変更を意味のある単位でコミットに分ける
コミット1: モデル定義を追加
コミット2: ビュー（画面）を追加
コミット3: テストを追加
→ 1つのPRにまとめてレビュー
```

**戦略2: PRそのものを分割する（大規模な場合）**
```
PR1: データベースのモデル定義
PR2: API エンドポイント（PR1がマージされてから）
PR3: フロントエンド画面（PR2がマージされてから）
```

**戦略3: フィーチャーフラグ（大幅な変更）**
新機能のコードは入れるが、フラグで無効化しておく。完成後にフラグをONにする。

### Q4: 戦略1と2の違い

`#コミット戦略` `#PR分割`

**質問**: 戦略1と2の違いを分かりやすく教えて。

**回答**:

```
戦略1:  1つの封筒に、付箋で区切った書類を入れる → まとめて提出
        （1 PR の中に複数のコミット）

戦略2:  封筒を3つに分けて、順番に提出する
        （PR を3つに分割）
```

| | 戦略1 | 戦略2 |
|--|-------|-------|
| PR数 | 1つ | 複数 |
| レビュー | まとめて1回 | PRごとに個別 |
| 向いている場面 | 中規模の機能 | 大規模・段階的にリリースしたい |
| リスク | レビューが大変になりうる | 依存関係の管理が必要 |

### Q5: 戦略1でコミット1の修正が必要になったら

`#コミット修正` `#作業フロー`

**質問**: 戦略1でコミット1の修正が必要となったら、ほかのコミットに影響は出ませんか？

**回答**:

**通常は影響が出ない。** 後のコミットは「差分の積み重ね」なので、コミット1の修正はコミット2以降にそのまま積み上がる。ただし、コミット1の変更を「なかったことにする」場合は、それに依存するコミットに影響が出る。

### Q6: 戦略1でコミット1を修正したら他のコミットに影響は出るか

`#コミット修正` `#履歴`

**回答**: Q5とほぼ同じ質問。修正内容が後のコミットと同じ箇所を触っていなければ影響なし。同じ箇所を触っている場合はコンフリクトの可能性がある。

### Q7: mainが先に進んでしまった時はどうすればいいか

`#merge` `#rebase` `#ブランチ同期`

**質問**: mainがどんどん先に進んでいる時は、どうすればいいのでしょうか？

**回答**:

2つの方法がある:

**方法1: `git merge main`（安全・初心者向け）**
```bash
git switch main
git pull origin main        # まず main を最新にする
git switch feature/xxx
git merge main              # main の変更を feature に取り込む
```

**方法2: `git rebase main`（履歴がきれい・中級者向け）**
```bash
git switch main
git pull origin main
git switch feature/xxx
git rebase main             # feature のコミットを main の先に付け替える
```

実務では方法1（merge）が安全で推奨。

### Q8: git rebase mainがよくわからない

`#rebase` `#履歴整理` `#merge比較`

**質問**: 方法2: `git rebase main`（履歴がきれい・中級者向け）がよくわからないです。

**回答**:

rebase は「re（再び）+ base（土台）」= **土台を置き直す**という意味。自分のコミットの分岐点を、mainの最新地点に「引っ越し」させる操作。

```
rebase 前:
main:     旧駅───A───B───新駅
                  \
feature:           ①───②         ← 「旧駅」から分岐

rebase 後:
main:     旧駅───A───B───新駅
                              \
feature:                       ①'──②'   ← 「新駅」から分岐し直した
```

**なぜ ①' ②' になるか**: 元の①②は「旧駅の時点のコード」を前提に書かれていたが、rebase後は「新駅の時点のコード」が前提になる。Gitが内部的にコミットをコピーして作り直すため、コミットIDが変わる。

**鉄則**: rebase は「まだ自分しか知らないコミット」に対して使う。push済みのコミットを rebase すると他の人の手元と食い違いが起きる。

### Q9: mergeとpullは何が違うのか

`#merge` `#pull` `#fetch` `#基礎概念`

**質問**: mergeをちゃんと理解できていないかもしれません。pullと何が違うの？

**回答**:

**`git pull` = `git fetch` + `git merge`。** pull は merge を**含んでいる**操作。

| 操作 | やること | 例えると |
|------|---------|--------|
| `fetch` | リモート（GitHub）の最新情報をダウンロード。作業ファイルには影響なし | 郵便受けを確認 |
| `merge` | 別ブランチの変更を今のブランチに合流させる | ノートに書き写す |
| `pull` | fetch + merge を一気にやるショートカット | 郵便受けから取って即反映 |

**merge の対象による違い:**

| コマンド | 何を | どこに | 場面 |
|---------|------|--------|------|
| `git pull origin main` | **GitHub上の** main を | 今のブランチに | リモートの変更を取得 |
| `git merge main` | **ローカルの** main を | 今のブランチに | ブランチ間の合流 |

## Step 2: Worktree vs Branch

### 概要

通常、1つのフォルダ（リポジトリ）では同時に1つのブランチしか展開できない。`git worktree` を使うと、**同じリポジトリのデータを使ってパソコン上に別のフォルダを作り、別ブランチを同時に開ける**。

### ブランチのみ vs Worktree

| | ブランチ（`git switch`） | ワークツリー（`git worktree`） |
|--|------------------------|------------------------------|
| **フォルダ数** | 1つ | 複数 |
| **切り替え方** | 中身が入れ替わる | 別フォルダを開く |
| **作業中のコード** | 退避が必要（stash等） | そのまま残る |
| **同時編集** | できない | できる（1人が複数ブランチを同時に開く） |
| **使い終わったら** | ブランチを削除 | `git worktree remove` でフォルダごと削除 |

### 基本コマンド

```bash
git worktree add <フォルダパス> -b <ブランチ名>   # 作成
git worktree list                                  # 一覧確認
git worktree remove <フォルダパス>                 # 削除
```

### 実践で体験したこと

1. `git worktree add ../git学習-feature -b feature/add-subtract` で別フォルダにブランチを展開
2. Worktree側で `app.py` に `subtract()` 関数を追加してコミット → push
3. `gh pr create` で PR #1 を作成
4. **元のフォルダ（main）は影響を受けていないことを確認**
5. squash merge で PR をマージ → `git pull origin main` でローカル更新
6. `git worktree remove` でクリーンアップ

### Q10: stashとは何か

`#stash` `#一時退避` `#基礎概念`

**質問**: stashとは何？

**回答**:

`git stash` は**作業中の変更を一時的に退避させる**機能。「机の上の書類を引き出しにしまう」イメージ。

```bash
git stash           # 編集内容を「引き出し」にしまう → ファイルが編集前に戻る
git switch main     # 安心してブランチを切り替えられる
git switch feature/xxx
git stash pop       # 「引き出し」から取り出す → 編集途中の状態に戻る
```

> Worktree を使えば stash が不要になるケースが多い。Step 3 で詳しく扱う。

### Q11: 同時編集とは一人が複数ブランチを編集できるという意味か

`#worktree` `#同時編集`

**質問**: ここでいう同時編集とは、複数人が同じフォルダを操作できるでなく、一人が複数ブランチを編集できるで認識があっている？

**回答**:

**その認識で完全に合っている。**

```
❌ 複数人が同じフォルダを操作できる
✅ 一人が複数ブランチを同時に開いて編集できる
```

具体的には VSCode ウィンドウを2つ開いて、それぞれ別フォルダ（別ブランチ）を同時に編集できる。複数人の共同編集は Git の機能ではなく、各自が自分のPCにリポジトリをクローンして、各自のブランチで作業 → PR でマージ（GitHub Flow）。

### Q12: pushでローカルのブランチがGitHubに登録された認識であっているか

`#push` `#リモート` `#ブランチ`

**質問**: Worktree側でコミット → push、ここではローカルにしかなかったブランチが、github側に登録されたで認識はあってる？

**回答**:

**完全に合っている。**

```
push 前: ローカルにだけ feature/add-subtract がある
push 後: GitHubにも feature/add-subtract が作られる
```

`git push -u origin feature/add-subtract` の各パーツ:
- `push`: ローカルの内容をGitHubに送る
- `-u`: ローカルブランチとリモートブランチを紐付ける（次回から `git push` だけで済む）
- `origin`: GitHubのリポジトリの名前
- `feature/add-subtract`: 送るブランチ名

## Step 3: git stash

### 概要

`git stash` は作業中の変更を一時的に退避させる機能。ブランチ切り替え時に未コミットの変更があると切り替えできない場合があるが、stash で退避すれば安全に切り替えられる。

### 基本コマンド

```bash
git stash                       # 変更を退避
git stash push -m "メモ"        # メモ付きで退避
git stash list                  # 退避リストの確認
git stash pop                   # 最新の退避を復元（リストから削除）
git stash apply                 # 最新の退避を復元（リストに残す）
```

### 実践で体験したこと

1. `feature/add-divide` ブランチで `divide()` 関数を途中まで書く
2. `git stash push -m "WIP: divide関数を作成中" -- app.py` で退避
3. `app.py` から `divide()` が消えたことを確認（引き出しにしまわれた）
4. `git stash pop` で復元 → `divide()` が復活

### stash vs Worktree の使い分け

| | stash | Worktree |
|--|-------|----------|
| **やりたいこと** | 一瞬だけブランチを切り替えたい | しばらく別ブランチで作業したい |
| **操作の重さ** | 軽い（2コマンドで完結） | やや重い（フォルダが増える） |
| **元の作業** | 見えなくなる（引き出しの中） | 別フォルダにそのまま残る |
| **向いている場面** | 「ちょっと確認して戻ってくる」 | 「別タスクを並行して進める」 |

> 実務の目安: 5分以内で戻れるなら stash、それ以上かかるなら Worktree が安全。

### Q14: stash向きの具体的な作業とは

`#stash` `#worktree` `#使い分け`

**質問**: 実務の目安: 5分以内で戻れるなら stash、とあるが、具体的にはどのような作業が対象？

**回答**:

**stash 向き（ちょっと見て戻ってくる）:**
- コードの確認（「mainのあの関数どう書いてあったっけ？」）
- 動作確認（「本番の状態でバグ再現するか試したい」）
- 設定のコピー（「別ブランチの設定ファイルを参考にしたい」）
- 急な1行修正（「READMEのタイポ直して」）

→ 共通点: **切り替え先で「見る」か「ほんの少し直す」だけ**

**Worktree 向き（しばらく別作業）:**
- 緊急バグ修正（調査・修正・テストで30分～）
- コードレビュー（他の人のPRをビルドして動作確認）
- 並行開発（機能Aのビルド待ちの間に機能Bを進める）
- 比較しながらの作業（旧新バージョンを並べて見比べる）

> Worktreeを覚えると stash をあまり使わなくなる人も多い。stash は「知っているけど場面を選ぶ」くらいの位置づけでOK。

### Q15: mainで修正した内容は元のブランチに反映されるか

`#merge` `#自動マージ` `#ブランチ間の同期`

**質問**: 急な1行修正など main を修正した場合、もとに戻ったブランチでは修正内容を反映させますか？

**回答**:

**自動では反映されない。だがほとんどの場合、何もしなくてOK。**

| 状況 | どうする |
|------|------|
| main の修正が自分の作業に**関係ない** | 何もしない。PRマージ時にGitが自動で両方を合流してくれる |
| main の修正が自分の作業に**影響する** | `git merge main` で取り込む |

> 「わざわざ取り込まなくても、最終的にマージする時に自動で合流する」のが Git の賢いところ。同じ箇所を触っている場合だけコンフリクトになる。
