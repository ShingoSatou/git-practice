## Step 4: コンフリクトの仕組み・回避・対処

### 概要

**コンフリクト（衝突）** とは、2つのブランチが「同じファイルの同じ行」を違う内容に変更した時、Git が「どっちを採用すればいいか分からない」状態になること。

### コンフリクトマーカーの読み方

```python
<<<<<<< HEAD
    return f"やあ、{name}！元気？"          # ← 自分（feature）の変更
=======
    return f"おはようございます、{name}さん！"  # ← main の変更
>>>>>>> main
```

| マーカー | 意味 |
|---------|------|
| `<<<<<<< HEAD` | ここから自分（今いるブランチ）の変更 |
| `=======` | 区切り線 |
| `>>>>>>> main` | ここまでが main の変更 |

### 解決方法

1. **自分の変更を採用** — HEAD側を残す
2. **main の変更を採用** — main側を残す
3. **両方を組み合わせて新しい内容にする** — 実務で一番多い

解決後: `git add <ファイル>` → `git commit` でマージ完了。

### 実践で体験したこと

1. main で `greet` 関数を「おはようございます」に変更してコミット
2. main の変更**前**の状態から `feature/change-greeting` ブランチを作成
3. feature で `greet` を「やあ」に変更してコミット
4. `git merge main` → 💥 コンフリクト発生！
5. マーカーを読み取り、両方を活かした「やあ、{name}さん！おはよう！」に修正
6. `git add` → `git commit` → 解決

### コンフリクトを避ける実務テクニック

| テクニック | 効果 |
|-----------|------|
| ブランチを小さく・短く保つ | mainとの差が小さくなり衝突しにくい |
| こまめに `git merge main` する | 差が広がる前に対処できる |
| チーム内で作業ファイルを分担する | 同じファイルの同時編集を減らす |
| PRのレビューを早くマージする | mainを常に最新に保てる |

### Q16: コンフリクトはどういう条件で発生するか

`#コンフリクト` `#発生条件`

**質問**: コンフリクトの条件を教えて。同じファイルの同じ場所をいじったとき？同じ関数をいじったとき？

**回答**:

**「同じファイルの同じ行」を2つのブランチが違う内容に変えた時。** Gitは「関数」や「意味」を理解しない。純粋に「行」の位置で判断。

| mainの変更 | featureの変更 | コンフリクト？ |
|------------|---------------|-------------|
| 2行目を変更 | 2行目を変更 | 💥 する（同じ行） |
| 2行目を変更 | 5行目を変更 | ✅ しない（違う行） |
| 同じ関数の3行目 | 同じ関数の5行目 | ✅ しない（同じ関数でも違う行ならOK） |

### Q17: commitの意味とpullとの違い

`#commit` `#pull` `#push` `#基礎概念`

**質問**: コミットの意味をあらためて教えて。pullとの違いは？

**回答**:

commitとpullは**方向が真逆**。

| 操作 | 方向 | 例えると |
|------|------|--------|
| `commit` | 自分のPC内でセーブ | ゲームのセーブ |
| `push` | 自分のPC → GitHub | セーブデータをクラウドにアップロード |
| `pull` | GitHub → 自分のPC | クラウドからダウンロード |

全体の流れ: `ファイル編集 → git add（準備） → git commit（セーブ） → git push（アップロード）`

### Q18: git addの意味

`#add` `#staging` `#基礎概念`

**質問**: git addの意味は？

**回答**:

**「次のコミットに含めるファイルを選ぶ」操作。** 変更したファイルのうち「これだけコミットしたい」と選ぶために存在する。

```
作業ディレクトリ        ステージ            リポジトリ（履歴）
 (ファイル編集)       (コミット準備中)    (セーブ済み)

  app.py     ──add──→  app.py     ──commit──→  記録！
  memo.txt              (選ばなかった)        (入らない)
```

| 操作 | 例えると |
|------|--------|
| ファイル編集 | 机の上で書類を書く |
| `git add` | 「この書類を提出する」と封筒に入れる |
| `git commit` | 封筒を閉じて提出（確定） |
| `git push` | 提出した封筒を郵便で送る（GitHubへ） |

## Step 5: rebase vs merge と履歴整理

### 概要

merge と rebase はどちらも「別ブランチの変更を取り込む」操作だが、履歴の残り方が違う。

### 3つの方式の比較

```
【通常 merge】           【squash merge】        【rebase + merge】
main: ●──●──M            main: ●──●──S           main: ●──●──①'──②'──③'
          / ↑ 合流              ↑ 1つにまとめ                ↑ 一直線
feature: ①②③                  (分岐の痕跡なし)              (IDが変わる)
```

| 方式 | main の履歴 | 分岐の記録 | コミット数 |
|------|------------|-----------|-----------|
| 通常 merge | 分岐あり | 残る | ブランチの全コミット＋マージコミット |
| **squash merge** | **一直線** | **残らない** | **1つだけ** |
| rebase + merge | 一直線 | 残らない | ブランチのコミット数分 |

### 実践で体験したこと

1. `feature/improve-app` ブランチで2つのコミット（divide関数追加 + 使用例追加）
2. mainにも別の変更（README更新）を入れて「分岐」状態を作る
3. `git rebase main` を実行 → 分岐が一直線になった
4. コミットIDが変わった（コピーして作り直されたため）

### Q19: ブランチを作成してから実装を始めるのか

`#ブランチ` `#ワークフロー`

**質問**: そもそもブランチを作成してから実装作業を開始する？それともコミットするときにブランチを作成するの？

**回答**:

**先にブランチを作ってから作業を始めるのが基本。**

```
✅ 正しい順番: git switch -c feature/xxx → 編集 → commit
❌ やりがち: mainのまま編集を始めてしまう
```

もし main で編集を始めてしまった場合、まだコミットしていなければ `git switch -c feature/xxx` で編集中の変更を持ったまま新ブランチに移れる。

> 「何か作業を始めよう」と思ったら `git switch -c` から始める。条件反射にする。

### Q20: 分岐がなくなることのメリットは

`#rebase` `#履歴` `#squash`

**質問**: 分岐がなくなることのメリットは？

**回答**:

主にチーム開発で効く。

| メリット | 説明 |
|---------|------|
| 履歴が読みやすい | `git log` で上から順に追える |
| バグ探しが楽 | `git bisect` で原因コミットを素早く特定できる |
| 差分が見やすい | レビュー時に変更の順番を追いやすい |

**実務の現実**: rebase を自分で使いこなす必要は薄い。GitHub の **squash merge** が同じ効果を自動で得られるので、日常的には squash merge に任せるのがスタンダード。

### Q21: squash mergeは合流ではなくまとめて新コミットを作る

`#squash` `#merge` `#履歴`

**質問**: squash merge は合流しているのは変わらないけど、1つのみの中継点になるイメージ？

**回答**:

**squash merge は「合流」ではない。** ブランチの全変更を**コピーして1つにまとめた新しいコミットを main に直接作る**操作。

```
通常 merge:  main: ●─●─M       ← M(マージコミット)で合流。分岐の記録が残る
                      /
feature:       ①②③

squash merge: main: ●─●─S       ← S = ①②③を全部まとめた新コミット。分岐の痕跡なし
feature:       ①②③  ← mainの履歴には現れない
```

**なぜ人気か**: main の履歴が「PR 1つ = コミット 1つ」でスッキリする。「何番のPRで何が入ったか」が一目瞭然。

## Step 6: コミットメッセージ規約（Conventional Commits）

### 概要

コミットメッセージに統一的なフォーマットを使うことで、履歴が読みやすくなり、自動化（変更ログの生成等）にも活用できる。

### フォーマット

```
<type>: <説明>

例: feat: ログイン機能を追加
```

### type の種類

| type | 意味 | 使う場面 |
|------|------|---------|
| `feat` | 新機能 | 関数追加・画面追加など |
| `fix` | バグ修正 | 動作の不具合を直した時 |
| `docs` | ドキュメント | README・コメントの修正 |
| `refactor` | リファクタリング | 動作は変えずコードを整理 |
| `test` | テスト | テストの追加・修正 |
| `chore` | 雑務 | 設定ファイル変更・依存関係更新 |
| `style` | スタイル | フォーマット修正（動作に影響なし） |
| `ci` | CI設定 | GitHub Actions等の変更 |

### 良い / 悪いメッセージの例

```
❌ 悪い例: 修正した / update / fix / aaa / WIP
✅ 良い例:
  feat: ユーザー認証にJWT方式を追加
  fix: ログイン時にパスワードが空でもエラーにならない問題を修正
  docs: APIエンドポイントのドキュメントを更新
```

### ブランチ名・コミット・PRの一貫性

```
ブランチ名:           feature/add-login
コミットメッセージ:    feat: ログイン機能を追加
PRタイトル:           feat: ログイン機能を追加
→ 全部が一貫していると、追跡しやすい
```
