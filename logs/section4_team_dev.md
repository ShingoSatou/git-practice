## Step 10: ブランチ保護ルール

### 概要

main ブランチに「保護ルール」を設定し、壊れないように守る仕組み。GitHub の Settings → Rules → Rulesets で設定する。

### 主な保護ルール

| ルール | 効果 | 個人開発 | チーム開発 |
|-------|------|---------|-----------|
| Require pull request | main への直接 push を禁止 | △ | ✅ 必須 |
| Require status checks | CI が通らないとマージ不可 | ✅ 推奨 | ✅ 必須 |
| Require approvals | レビュー承認がないとマージ不可 | ✗ 不要 | ✅ 必須 |
| Restrict force push | `git push --force` を禁止 | ✅ 推奨 | ✅ 必須 |

### 料金

- パブリックリポジトリ: 無料
- プライベートリポジトリ: GitHub Pro（月$4）が必要

### AI駆動開発での位置づけ

チームに参加した時、「main に直接 push できない」のはバグではなくブランチ保護ルール。PR 経由でマージするのが正しい手順。

## Step 11: PR / Issue テンプレート

### 概要

GitHub では、PR や Issue の作成時に**テンプレート**を用意できる。テンプレートがあると、書くべき情報の抜け漏れが減り、チーム全体で統一されたフォーマットになる。

### テンプレートの配置場所

| テンプレート | パス |
|------------|------|
| PR テンプレート | `.github/pull_request_template.md` |
| Issue テンプレート（単一） | `.github/ISSUE_TEMPLATE.md` |
| Issue テンプレート（複数） | `.github/ISSUE_TEMPLATE/*.md` |

### 実践で作成したファイル

```
.github/
├── ISSUE_TEMPLATE/
│   ├── bug_report.md        ← バグ報告テンプレート
│   └── feature_request.md   ← 機能要望テンプレート
├── pull_request_template.md ← PR テンプレート
└── workflows/
    └── ci.yml               ← CI（既存）
```

### ポイント

- Issue テンプレートの **frontmatter**（`---` で囲まれた部分）で `name`, `about`, `labels` を指定 → 作成時の選択画面に表示される
- テンプレートは **main ブランチにマージされて初めて** GitHub 上で有効になる
- `gh pr create` でも PR テンプレートが自動適用される

### AI駆動開発での位置づけ

AI に PR を作らせるときも、テンプレートがあれば統一されたフォーマットで出力される。人間・AI 問わず品質が揃う。

## Step 12: タグ・リリース

### 概要

Git の**タグ**は、特定のコミットに「名札」を付ける機能。主にバージョン番号（`v1.0.0`）の記録に使う。GitHub の**リリース**は、タグにリリースノートや配布物を添付する機能。

### タグの種類

| 種類 | コマンド | 特徴 |
|------|---------|------|
| 軽量タグ | `git tag v1.0.0` | 単なるポインタ |
| 注釈付きタグ | `git tag -a v1.0.0 -m "msg"` | 作成者・日時・メッセージ付き（**推奨**） |

### セマンティックバージョニング（SemVer）

```
v MAJOR.MINOR.PATCH
  破壊的変更 . 機能追加 . バグ修正
```

- MAJOR: 後方互換性を壊す変更（1.x → 2.0）
- MINOR: 後方互換性のある機能追加（1.0 → 1.1）
- PATCH: バグ修正のみ（1.0.0 → 1.0.1）

### よく使うコマンド

| コマンド | 効果 |
|---------|------|
| `git tag -l` | タグ一覧 |
| `git show v1.0.0` | タグの詳細 |
| `git tag -d v1.0.0` | タグ削除（ローカル） |
| `git push origin v1.0.0` | タグを push |
| `git push origin --tags` | 全タグをまとめて push |

### 注意点

- `git push` だけではタグは送られない → 明示的に `git push origin --tags` が必要
- GitHub リリースは CLI: `gh release create v1.0.0 --title "v1.0.0" --notes "..."` でも作成可能

### 実践

- `v1.0.0` を注釈付きタグとして作成した

### AI駆動開発での位置づけ

CI/CD でタグ push をトリガーに自動デプロイできる（`on: push: tags: ['v*']`）。タグを push するだけで本番リリースが走る仕組みが一般的。

### Q&A

**Q: リリースとはどんな機能？**

A: GitHub の**リリース**は、タグ（バージョン）に対して**リリースノートや配布物を添付して公開**できる機能。

- **タグ** = Git の機能（コミットに名札を付ける）
- **リリース** = GitHub の機能（タグに説明文・ファイルを添えてユーザーに公開する）

リリースに含められるもの：
- リリースノート（変更内容の説明）
- 添付ファイル（ビルド済みバイナリ、ZIP など）
- 自動生成ノート（GitHub がマージ済み PR 一覧から自動作成）

作成の流れ：
```bash
git tag -a v1.0.0 -m "msg"    # タグを作る（ローカル）
git push origin v1.0.0         # タグをリモートに送る
gh release create v1.0.0       # リリースとして公開する
```

タグだけでもバージョン管理はできるが、リリースを作ることで「何が変わったのか」を分かりやすく伝えられる。
